## 1. 项目概览

### 1.1 背景

香港建筑及室内设计行业投标过程中，成本计算占整个流程 70-80%，依赖手动阅读需求说明、收集报价、Excel 输入。系统旨在解决「找商机、读标书、写标书」三大环节。

### 1.2 目标

建立一套 AI 驱动的投标辅助系统，包含：

- **优化机会筛选**：自动搜寻、AI 匹配、商机推送
- **缩短成本计算时间**：图纸与项目清单并列、表单化输入
- **提升准确性**：自动化数量对照与价格计算
- **标书自动生成**：AI 生成完整标书文档
- **标书自动审查**：完整性、一致性、合规性检查
- **多端支持**：桌面 + 手机 PWA + 语音交互
- **IFC/BIM 集成**：支持 IFC (ISO 16739) 文件解析、3D 模型查看、BOQ 自动提取、IFC 文件交付

### 1.3 目标用户

| 角色 | 说明 |
|------|------|
| 管理层 | 审核标书、筛选投标机会、设定利润率与最终定价 |
| 投标负责人 | 成本估算、文件整合、标书填写、审核提交 |

### 1.4 成功指标

| 指标 | 目标值 |
|------|--------|
| 成本计算时间缩短 | 减少 30% 以上 |
| 输入错误率降低 | 降低 50% |
| 使用者满意度 | 平均 4 分以上（1-5 分）|
| 系统采用率 | 80% 以上投标案使用系统辅助 |
| 投标机会筛选效率 | 筛选时间减少 50% 以上 |

### 1.5 设计决策摘要

| 决策项 | 选择 | 备选 |
|--------|------|------|
| 架构模式 | 微服务 | 一体化 / Serverless |

## 3. AI 智能体设计

### 3.1 智能体核心架构

AI 智能体服务基于 LangGraph 构建，包含三个核心组件：

**Agent Orchestrator（智能体编排器）**

- **Planner（规划器）**：解析用户意图，确定需要调用的 Skills
- **Executor（执行器）**：按计划依次调用 Skills，处理工具返回
- **Response Formatter（回应格式化器）**：将结果格式化为结构化输出
- **Skill Router（技能路由器）**：根据任务类型分派到对应 Skill

**LLM Provider Layer（Vercel AI SDK）**

- 统一接口支持 OpenAI GPT-4o、Anthropic Claude、Qwen/DeepSeek、本地模型（Ollama）
- 配置文件驱动切换，支持 Fallback 链
- 成本追踪与用量统计

**RAG Pipeline**

- 文档摄入 → 分块（Chunking）→ Embedding 生成 → pgvector 存储 → 语义检索 → 上下文增强生成

### 3.2 Skill Registry（8 个 Skills）

| Skill | 输入 | RAG 检索源 | 输出 |
|-------|------|-----------|------|
| 1. 标书配对 | 招标公告 PDF + 公司能力档案 | 历史标书、过往匹配记录 | 匹配度 0-100 + 理由 + 建议行动 |
| 2. 备注生成 | 标书条款 + 项目资讯 | 技术方案库、历史标书同类备注 | 专业备注文本（含技术参数）|
| 3. 风险评估 | 招标文件全文 | 招标规范、行业标准、合规要求 | 风险标签列表 + 应对建议 |
| 4. 价格分析 | 项目清单 + 物料需求 | 物料报价数据、历史同类单价 | 建议单价范围 + 替代物料 |
| 5. 图纸解析 | PDF/DWG + 项目清单 | 历史图纸-项目映射 | 图纸-项目关联 + 定位标记 |
| 6. 标书生成 | 项目资讯 + 成本表 + 招标要求 | 历史标书模板、技术方案、中标文档 | 完整标书 PDF（封面到附件）|
| 7. 标书审查 | 已完成标书 + 原始招标要求 | 招标规范、评分准则 | 审查报告（缺失 + 不一致 + 评分预估）|
| 8. IFC/BIM 分析 | IFC 文件 (ISO 16739) | 历史 IFC 模型、BIM 规范 | BOQ 清单 + 合规检查 + IFC 交付文件 |

### 3.3 知识领域

RAG 系统维护五个知识领域：

1. **历史标书库**：中标/失败标书、评标反馈、成功策略
2. **技术方案库**：施工方案、安全措施、工艺描述、标准文本
3. **物料报价数据库**：物料规格、供应商报价、替代方案
4. **招标规范文件库**：政府招标公告、规范文件、行业标准
5. **业务操作流程**：公司投标流程、经验知识、决策逻辑

## 5. 业务流程

### 5.1 FR-01 商机筛选流程（全自动）

1. 爬虫服务定时扫描目标站点 → 发现新招标公告
2. 自动下载招标 PDF → 发消息到 `tender.raw` RabbitMQ 队列
3. AI 智能体接收 → 调用**图纸解析 Skill** 提取结构化资讯
4. 调用**标书配对 Skill** → RAG 检索历史标书 + 公司能力 → 生成匹配度评分
5. 调用**风险评估 Skill** → 扫描风险条款 → 标记风险标签
6. 招标项目服务储存结果 → 生成「商机决策卡」
7. 通知服务推送 Email 给管理层

### 5.2 FR-02/03/04 成本计算流程（用户操作）

1. 投标负责人打开招标项目 → 创建新 Bid
2. 上传图纸（PDF / DWG / IFC）→ Document Service / BIM Service 解析
   - a. **PDF**：PDF.js 渲染，页面跳转 + 区域高亮
   - b. **DWG**：服务端转 PDF 后查看
   - c. **IFC**：BIM Service 解析 → 提取建筑元素 → 自动生成 BOQ 清单 → 3D 模型查看器展示
3. 左右分屏：左侧项目清单表单 · 右侧图纸/3D 模型查看器
4. 点选项目清单中的某项 → 图纸跳转并高亮 / 3D 模型中高亮对应构件
5. 输入数量、成本价格 → 可从物料资料库快速选取
   - a. **IFC 项目**：数量由 IFC 模型自动提取，用户确认即可
6. AI **价格分析 Skill** 即时检索历史报价，提供定价建议
7. 设定利润率 → 系统自动计算各项售价 + 总投标价
8. 调用**标书生成 Skill** → 生成完整标书 PDF
9. 调用**标书审查 Skill** → 生成审查报告，列出修正项
10. 确认无误 → 一键导出（PDF / Excel / IFC 交付文件）

### 5.3 服务间通信模式

| 模式 | 场景 | 技术 |
|------|------|------|
| 同步 REST | 前端 CRUD、查询 | HTTP/JSON via API Gateway |
| 异步消息 | 爬虫 → AI → 项目 → 通知 | RabbitMQ |
| WebSocket | AI 分析进度、语音流 | Socket.io |

## 7. 新增功能

### 7.1 标书生成（Skill 6）

**流程：**

1. 读取招标要求 → 确定标书章节结构
2. 逐章节调用备注生成 Skill 填充内容
3. RAG 检索同类中标标书作为参考
4. 自动填入公司资讯、业绩描述、人员资历
5. 多页一致性校验 → 生成打印就绪 PDF

**输出：** 完整标书文档，包含封面、目录、各章节、报价表、附件清单。

### 7.2 标书自动审查（Skill 7）

**流程：**

1. 对照招标要求逐项检查响应完整性
2. 检测多页数据一致性（人名、牌照、金额）
3. 识别缺失文件/签名/附件
4. 模拟评审打分（若评分准则可获取）
5. 生成清单式审查报告 + 修正建议

**输出：** 审查报告，包含缺失项列表、不一致警告、风险提示、评分预估。

### 7.3 IFC/BIM 集成（ISO 16739）

IFC (Industry Foundation Classes) 是建筑信息模型 (BIM) 的国际开放标准（ISO 16739）。系统支持完整的 IFC 工作流：解析、查看、数量提取、文件交付。

**Skill 8：IFC/BIM 分析**

- **输入：** IFC 文件 (ISO 16739) + 招标要求
- **RAG 检索：** 历史 IFC 模型分析结果、BIM 规范标准
- **输出：** BOQ 自动清单 + 模型合规检查 + IFC 交付文件

**核心能力：**

| 能力 | 描述 | 技术 |
|------|------|------|
| IFC 解析 | 读取 IFC 文件，提取建筑元素、空间结构、材料属性 | IfcOpenShell (Python) |
| BOQ 自动提取 | 从 IFC 模型自动提取工程量清单（墙面积、地板面积、门窗数量等）| IfcOpenShell + 自定义规则引擎 |
| 3D BIM 查看 | Web 端 3D 模型渲染，支持构件选取、属性查看、高亮定位 | @thatopen/components (web-ifc) |
| 构件-项目关联 | IFC 构件与标书项目清单双向关联，点选即定位 | 前端状态管理 + 映射表 |
| IFC 合规检查 | 检查 IFC 模型是否符合招标要求（尺寸、材料、规格）| AI Agent + IfcOpenShell |
| IFC 文件交付 | 根据标书数据生成/修改 IFC 文件，作为投标交付物 | IfcOpenShell 写入 + 属性更新 |
| IFC 属性扩展 | 在 IFC 模型中添加投标相关属性（成本、工期、厂商）| IfcOpenShell property set 操作 |
| 3D Geometry | 提取几何数据，生成 Web 端可渲染的格式（glTF/JSON）| — |

**IFC 工作流：**

```
上传 IFC 文件
    │
    ▼
BIM/IFC 服务解析 (IfcOpenShell)
    ├── 提取建筑元素 → 生成元素树
    ├── 计算工程量 → 自动生成 BOQ 清单
    ├── 提取材料规格 → 匹配物料资料库
    └── 提取空间结构 → 生成楼层/区域索引
    │
    ▼
前端 3D BIM 查看器渲染
    ├── 构件选取 → 显示属性面板
    ├── 点选标书项目 → 3D 模型中高亮对应构件
    └── 缩放/旋转/剖面查看
    │
    ▼
AI 分析
    ├── IFC/BIM 分析 Skill → 合规检查 + 风险识别
    └── 价格分析 Skill → 匹配历史同类 IFC 项目报价
    │
    ▼
交付阶段
    ├── 更新 IFC 属性（成本、厂商、规格）
    ├── 导出 IFC 交付文件
    └── 连同 PDF/Excel 标书一起提交
```

**IFC 服务架构：** 新增独立的 BIM/IFC 服务（Python），基于 IfcOpenShell：

- **IFC Parser：** 解析 IFC4/IFC4x3 文件，提取 IfcProduct、IfcQuantityArea、IfcQuantityLength 等
- **BOQ Engine：** 按规则将 IFC 元素映射为工程量清单项（如 IfcWall → 墙面面积，IfcDoor → 门数量）
- **IFC Writer：** 修改 IFC 文件属性、添加 Pset（Property Set）、生成交付文件

| 场景 | 描述 | 技术 |
|------|------|------|
| 语音对话 | 外出时口语查询商机资讯 | Whisper STT → AI Agent → Edge TTS |
| 语音备注 | 口述备注，AI 转文字并润色 | Whisper STT → 备注生成 Skill |
| 语音播报 | AI 主动播报商机摘要和风险提示 | Edge TTS |
| 会议记录 | 投标讨论会议语音转文字，提取行动项 | Whisper STT → 结构化提取 |

**语音技术栈：**

- **STT：** OpenAI Whisper API（支持粤语 + 普通话 + 英文）
- **TTS：** Edge TTS（免费）+ OpenAI TTS（备选）
- **前端：** Web Audio API + MediaRecorder（浏览器原生录音）
- **实时流：** WebSocket 边说边识别，低延迟回应，支持打断

## 8. 部署架构

### 8.1 生产环境

```
CloudFlare CDN (SSL + DDoS 防护)
    │
    ▼
Kubernetes Cluster
    │
    ├── Ingress (Nginx)
    │   ├── / → Frontend (Next.js ×2)
    │   └── /api/* → API Gateway (Node.js ×2)
    │
    ├── Microservices (各服务独立 Pod)
    │   ├── User Service ×1
    │   ├── Tender Project Service ×2
    │   ├── Bid Service ×2
    │   ├── Scraper Service ×1
    │   ├── AI Agent Service ×2 (GPU)
    │   ├── BIM/IFC Service ×1
    │   ├── Notify Service ×1
    │   └── Voice Service ×1
    │
    └── Data Layer
        ├── RDS PostgreSQL (主从复制)
        ├── ElastiCache Redis
        ├── S3 Bucket (文件存储)
        └── pgvector (向量存储，PostgreSQL 扩展)
```

### 8.2 开发环境

Docker Compose 一键启动所有服务，包含本地 PostgreSQL、Redis、RabbitMQ、MinIO。

### 8.3 CI/CD

GitHub Actions 流水线：

- PR 触发：Lint + Unit Test + Integration Test
- Merge to main：Docker Build → Push to ECR/ACR → Deploy to Staging
- 手动 Approval → Deploy to Production
- 每个微服务独立 repo，独立 CI/CD pipeline

## 附录 B：RAG 文档摄入流程

> **注：** 文档预处理（PDF/Word/Excel/OCR）由 **BIM/IFC Python 服务**承担，与 IFC 文件处理共用同一 Python 环境，保持 Python 仅用于文件处理领域。

```
用户上传文件 (PDF/Word/Excel)
    │
    ▼
文档预处理（BIM/IFC Python 服务）
    ├── PDF → PyMuPDF 提取文字 + 表格
    ├── Word → python-docx 提取结构
    ├── Excel → pandas 解析数据表
    └── 图纸 → OCR (Tesseract) 提取标注
    │
    ▼
智能分块
    ├── 按章节/段落分块
    ├── 保留元数据（来源、页码、项目关联）
    └── 每块 500-1000 tokens
    │
    ▼
Embedding 生成（AI 智能体服务 Node.js 调用 API）
    ├── 使用 text-embedding-3-large (OpenAI)
    └── 或本地 BGE-M3 模型
    │
    ▼
存入 pgvector
    ├── 向量 + 元数据 + 原文
    └── 按知识领域分区索引
```
